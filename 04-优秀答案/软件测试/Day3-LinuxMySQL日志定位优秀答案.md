## 1. 接口返回 500 如何结合 Linux 日志和 MySQL 定位

### 结论

接口 500 要先固定失败现场, 再用 traceId / 时间戳查 Linux 日志, 结合 MySQL 数据状态、死锁、慢 SQL 和环境配置归因。

### 优秀答案

以数据分析平台接口自动化回归为例, 如果发现某个接口返回 500, 我会先固定失败现场, 再到 Linux 和 MySQL 侧做定位。

第一步确认现象。先记录接口 URL、请求方法、请求头、token、入参、响应体、HTTP 状态码、业务错误码、执行环境、失败时间点。如果响应里有 traceId 或 requestId, 我会优先记录下来, 后续用它串联日志。

第二步查应用日志。登录 Linux 服务器后, 进入对应服务日志目录, 先按时间点和关键字过滤日志。例如用 `grep` 查 ERROR、Exception、traceId、接口路径等关键字。如果日志量很大, 可以用 `less` 查看上下文, 或用 `tail -n` 看最近日志。如果问题能复现, 我会用 `tail -f` 实时观察日志, 同时重新发起接口请求, 确认该请求对应的完整异常堆栈。

第三步分析日志内容。看 500 是空指针、参数解析异常、SQL 执行异常、唯一键冲突、连接超时、依赖服务不可用, 还是权限数据异常。比如日志里出现 connection refused, 就偏向依赖服务或数据库连接问题；如果是 SQL syntax 或 duplicate key, 就偏向后端 SQL 或数据问题；如果是 NullPointerException, 就要结合入参和数据状态看是否有空值未处理。

第四步做 MySQL 验证。如果接口依赖某条业务数据, 我会按请求参数查数据库, 看数据是否存在、状态是否满足前置条件、用户是否有权限、关联表是否缺数据。如果怀疑并发回归导致死锁, 会用 `SHOW ENGINE INNODB STATUS` 查看最近死锁信息, 比对死锁时间和接口失败时间。如果怀疑慢查询或连接池问题, 还会关注慢 SQL、连接数和数据库日志。

第五步归因。如果日志异常和数据库状态都能对应上, 我会把问题归到代码缺陷、数据问题、数据库问题、依赖服务问题或环境配置问题。比如测试数据状态不满足前置条件, 可能是自动化数据准备或清理问题；如果同样数据手工复现稳定 500, 且日志指向代码异常, 就可以按接口缺陷提交。

最后提交缺陷时, 我会附上接口 URL、请求参数、响应体、失败时间、traceId、日志关键堆栈、数据库验证结果、复现步骤和影响范围。这样开发可以直接定位, 也方便后续把该场景沉淀为回归用例。

### 常见追问

- 日志没有 traceId 怎么办?
- 500 只在 Jenkins 出现怎么定位?
- 如何判断是数据问题还是代码缺陷?

### 易错点

- 只看接口响应, 不保留失败时间和 traceId。
- 只查日志, 不验证数据库状态。

## 2. 日志只有 NullPointerException 时如何继续定位

### 结论

NullPointerException 要看完整上下文、堆栈顶层业务类、方法名和行号, 再结合入参、数据库、依赖服务和配置判断空对象来源。

### 优秀答案

如果日志里只看到 `NullPointerException`, 我不会只停留在异常名称, 会继续把完整上下文拿出来分析。

首先我会按失败时间点、traceId 或接口路径把完整日志上下文查出来, 不只是 grep 一行。比如查看异常前后的请求参数、用户信息、接口路径、调用链和完整堆栈。重点看堆栈最上面的业务类、方法名和行号, 因为那里通常能判断是哪一个对象为空。

然后我会结合接口入参和数据库数据判断空值来源。常见原因有几类: 入参缺少必填字段但后端没有校验；数据库查询结果为空但代码直接取字段；关联表缺数据；依赖服务返回空对象；配置项或缓存没有初始化。比如报表查询接口如果某个部门 ID 查不到配置, 后端又直接读取配置对象字段, 就可能出现空指针。

接着我会做复现和缩小范围。用同样账号、同样请求参数、同样测试数据重新调用接口, 看是否稳定复现。如果稳定复现, 我会调整单个参数或补齐关联数据, 判断是哪类数据触发了空指针。如果只有 Jenkins 批量回归时出现, 我会再检查是否有数据清理、并发执行、缓存失效或环境配置问题。

如果需要开发本地 debug, 我会把请求参数、测试数据 ID、时间点、traceId、日志堆栈、数据库查询结果提供给开发, 让开发用相同分支和相同数据复现。测试侧可以给出初步怀疑方向, 比如“某个关联表查询为空后未做判空”, 但最终根因由代码定位确认。

缺陷单里我会写清楚: 触发接口、入参、复现步骤、完整堆栈、怀疑的空对象来源、数据库验证结果和影响范围。这样开发能快速定位, 也方便修复后补充异常参数和缺失数据的回归用例。

### 常见追问

- 如何判断空对象来自入参还是数据库?
- 偶现 NPE 怎么缩小范围?
- 测试如何协助开发 debug?

### 易错点

- 只 grep 异常名, 不看完整堆栈和业务行号。
- 只说本地 debug, 不提供可复现证据。

## 3. Jenkins 批量回归才出现 500 如何排查

### 结论

批量回归才出现 500, 优先怀疑并发、数据污染、用例依赖、异步未完成、限流、连接池不足和环境差异。

### 优秀答案

如果 500 只在 Jenkins 批量回归时出现, 单独调用接口不复现, 我会优先怀疑批量执行带来的环境、数据和并发问题, 而不是直接判定接口逻辑错误。

第一类是并发和数据库问题。比如多个用例同时写同一类数据, 可能出现死锁、唯一键冲突、事务等待或连接池耗尽。我会通过失败时间点去查应用日志和数据库日志, 如果怀疑死锁, 用 `SHOW ENGINE INNODB STATUS` 查看最近死锁信息, 对比 Jenkins 失败时间和 SQL 语句。

第二类是用例依赖和数据污染。Jenkins 批量跑时, 用例执行顺序可能和本地不一样, 如果某个用例依赖另一个用例创建的数据, 或者某个清理逻辑把其他用例还在用的数据删了, 就会导致偶现 500。我会查看失败用例前后执行了哪些用例, 对比测试数据 ID, 检查是否有共享数据、共享账号、重复设备编号或数据被提前清理。

第三类是异步任务或数据未就绪。比如某个接口创建报表后, 下游统计任务还没完成, Jenkins 紧接着调用查询接口, 就可能查不到预期数据或触发异常。我会看各接口调用时间戳、任务状态、数据库更新时间, 必要时增加状态轮询而不是固定 sleep。

第四类是资源和限流问题。Jenkins 批量执行可能带来高并发请求, 导致接口限流、服务线程池满、数据库连接池不足、响应超时。我会结合服务端日志、网关日志、接口耗时、错误码和资源监控确认。

第五类是环境差异。确认 Jenkins 使用的 base_url、配置文件、账号、依赖版本、数据库是否和本地一致, 防止 Jenkins 指向了共享测试环境或预发布环境。

验证方式上, 我会先保留失败现场, 包括构建号、失败时间、请求参数、响应体、traceId、数据 ID；然后在 Jenkins 上单独重跑该用例、多次循环跑、关闭并发跑、使用独立测试数据跑。如果关闭并发后稳定通过, 更偏向并发或数据冲突；如果固定同一请求仍然偶现 500, 就更像服务端稳定性问题。

### 常见追问

- 如何证明是数据污染?
- 如何证明是并发导致?
- 为什么不能直接说是后端缺陷?

### 易错点

- 只怀疑死锁, 没有覆盖用例依赖、数据清理和限流。
- 没有在 Jenkins 上做隔离复现。

## 4. 报表数量不准如何用 MySQL 验证

### 结论

报表数量不准要先统一页面、接口和 SQL 的筛选条件, 再验证 count、group by、join、权限范围、数据来源和时效性。

### 优秀答案

如果报表接口返回 200, 但统计数量明显不对, 我会先确认页面展示、接口返回和数据库统计三方的口径是否一致。

第一步先固定查询条件。包括时间范围、状态、部门、用户角色、租户、数据权限、是否排除逻辑删除、是否只统计有效数据。不能只看页面上选了什么, 还要看接口实际传了什么参数。

第二步用 MySQL 按同样条件查明细数量。比如接口返回某个状态下报表数量为 100, 我会用类似 `select count(*) from table where status = ? and create_time between ? and ? and is_deleted = 0` 的思路验证。这里重点不是只写 count, 而是确认 where 条件和接口统计口径完全一致。

第三步验证分组统计。如果页面展示的是各状态数量, 我会按状态 `group by status` 查每类数量, 看具体是哪一类偏差。这样比只查总数更容易定位问题。

第四步验证联表逻辑。如果报表统计涉及多表关联, 我会检查 join 条件是否正确, 有没有一对多导致重复统计, 是否需要 `count(distinct id)`, inner join 是否把无关联记录过滤掉, left join 是否引入了脏数据。很多报表数量不准都出在关联和去重上。

第五步验证权限范围。如果当前登录用户只能看自己部门或自己创建的数据, 手写 SQL 也要加上相同的数据权限条件, 否则数据库查全量, 接口查权限范围内数据, 数量自然不一致。

第六步确认数据来源和时效性。报表可能不是实时查明细表, 而是读汇总表、缓存表或定时任务结果。如果页面读汇总表, 我手写 SQL 查明细表, 就要确认汇总任务是否已执行、缓存是否刷新、统计时间点是否一致。

第七步确认环境和 schema。确保接口连接的数据库、schema、租户和我手工查询的是同一套环境, 避免测试环境、预发布环境或库表连接错导致误判。

最后根据对比结果归因: 如果数据库按同口径查询结果和接口不一致, 偏后端 SQL 或统计逻辑问题；如果接口和数据库一致但页面不一致, 偏前端字段映射或展示问题；如果汇总表和明细表不一致, 就要查同步任务或缓存刷新问题。

### 常见追问

- 只查 count 为什么不够?
- 权限范围为什么会影响统计数量?
- 汇总表和明细表不一致怎么处理?

### 易错点

- 只写 `select count(*)`, 不统一统计口径。
- 忽略权限、缓存、汇总表和定时任务。

## 5. 多表 join 导致统计变大如何判断

### 结论

判断 join 放大要先确认主统计对象, 再逐步加 join, 用主键分组和 `having count(*) > 1` 查重复。

### 优秀答案

如果多表 join 后统计结果比预期大很多, 我会重点怀疑一对多关联把主表记录放大了。

第一步我会先确认主统计对象是什么。比如报表数量统计的是主表里的报表记录, 那最终就应该按主表 id 计数, 不能因为关联了明细表、标签表、权限表后把一条主记录变成多行。

第二步我会把复杂 SQL 拆开看。先单独查主表在同样筛选条件下有多少条, 再逐步加 join, 每加一张表就看 count 是否突然变大。如果加到某张子表后数量变大, 说明这张表和主表可能是一对多关系。

第三步我会查重复主键。比如按主表 id 分组, 看哪些主记录 join 后出现多次:

```sql
select main.id, count(*)
from main_table main
join child_table child on child.main_id = main.id
where ...
group by main.id
having count(*) > 1;
```

如果能查出很多主表 id 对应多条子表记录, 就基本能确认是一对多关联导致重复统计。

第四步看 join 条件是否正确。有时不是正常一对多, 而是 join 条件漏了租户、状态、版本、时间范围等条件, 导致关联了不该关联的数据。

第五步修正验证 SQL。常见修正方式有几种: 如果统计的是主表数量, 可以用 `count(distinct main.id)`；如果只需要判断子表是否存在, 可以用 `exists`；如果需要子表统计值, 可以先对子表按主表 id 聚合成一行, 再 join 主表；如果是 join 条件漏条件, 就补齐关联条件。

最后把修正后的 SQL 结果和接口返回、页面展示对齐。如果修正后数量正常, 就可以把问题定位为后端统计 SQL 的 join 放大或去重缺失问题。

### 常见追问

- distinct 能解决所有重复统计吗?
- exists 适合什么场景?
- join 条件漏租户会产生什么问题?

### 易错点

- 只说加 distinct, 不验证主键重复原因。
- 不确认主统计对象。

## 6. Linux 常用排查命令分别适合什么场景

### 结论

Linux 命令要按排查场景使用: 实时日志、关键字过滤、大文件查看、历史压缩日志、进程端口、系统资源、只读配置确认。

### 优秀答案

Linux 上排查接口问题, 我常用命令会按场景分。

第一类是查看实时日志。接口可以复现时, 我会用:

```bash
tail -f app.log
```

或者先看最近日志:

```bash
tail -n 200 app.log
```

适合一边发起接口请求, 一边观察服务端是否打印异常、错误码、SQL 报错或 traceId。

第二类是按关键字过滤日志。比如接口返回 500, 我会按时间、traceId、接口路径或异常关键字查:

```bash
grep "traceId" app.log
grep -n "ERROR" app.log
grep -n "NullPointerException" app.log
```

如果要看异常上下文, 可以用:

```bash
grep -A 50 -B 20 "NullPointerException" app.log
```

这样能看到异常前后的请求参数、调用链和堆栈。

第三类是查看大文件日志。日志很大时, 不适合直接 `cat`, 我会用:

```bash
less app.log
```

在里面搜索关键字。`cat` 更适合小文件, 大日志用 `cat` 会刷屏。

第四类是查找日志文件。比如不知道日志在哪个目录, 可以用:

```bash
find /data/logs -name "*.log"
find /data/logs -name "*error*"
```

如果日志按日期压缩, 可以用 `zgrep` 查历史压缩日志:

```bash
zgrep "traceId" app.log.2026-06-21.gz
```

第五类是查看进程和端口。如果接口连接不上或服务不可用, 我会查服务进程和端口:

```bash
ps -ef | grep service_name
netstat -tulnp | grep 8080
```

确认服务是否启动、端口是否监听。

第六类是查看系统资源。如果接口大量超时或 Jenkins 批量回归失败, 我会看 CPU、内存、磁盘:

```bash
top
free -m
df -h
```

确认是不是资源不足、磁盘满或服务负载过高。

第七类是查看配置。比如怀疑连接了错误环境, 我会只读查看配置文件:

```bash
grep "mysql" application.yml
grep "spring.profiles.active" application.yml
```

但不会随意在服务器上用 `vim` 修改配置。测试发现配置问题后, 应按变更流程让对应负责人修改。

所以我不会只是记命令, 而是按接口 500、超时、连接失败、数据不一致、服务未启动、资源不足这些场景选择对应命令排查。

### 常见追问

- 大日志为什么不用 cat?
- 如何查压缩日志?
- 测试为什么不能随意 vim 改服务器配置?

### 易错点

- 只列命令, 不说场景。
- 把编辑服务器文件当成常规排查动作。

## 7. 接口响应慢如何区分服务、SQL、网络或依赖问题

### 结论

接口慢要先定位耗时发生在哪一段, 再结合服务日志、Linux 资源、MySQL 执行计划、下游调用和网络耗时区分原因。

### 优秀答案

如果接口响应很慢但没有报 500, 我会先确认慢在哪里, 再区分是服务本身、SQL、依赖服务还是网络问题。

第一步先固定现象。记录接口 URL、请求参数、响应时间、发生时间、traceId、是否所有参数都慢, 还是某类查询条件慢。比如报表查询接口, 如果只在大时间范围或某个部门查询慢, 很可能和数据量或 SQL 有关。

第二步看服务端日志。很多服务会打印接口耗时或调用链耗时, 我会用 traceId 查应用日志, 看总耗时里是 controller、业务处理、数据库查询还是调用下游服务耗时长。如果日志里有 SQL 耗时或外部调用耗时, 可以直接缩小范围。

第三步看 Linux 资源。如果服务整体变慢, 我会用 `top` 看 CPU, `free -m` 看内存, `df -h` 看磁盘, 还可以看服务进程是否占用异常。如果 CPU 打满、内存不足、磁盘满或频繁 GC, 更偏服务资源问题。

第四步查 MySQL。如果怀疑 SQL 慢, 不只是手工执行 SQL, 还要看慢查询日志、执行计划和索引。可以用 `EXPLAIN` 看是否走索引、扫描行数是否过大、是否有 filesort 或临时表。也要看是否存在锁等待、连接数过高或大事务未提交。如果接口慢只发生在特定筛选条件, 就要重点分析这个条件对应的 SQL 和数据量。

第五步看依赖服务。如果该接口需要调用其他系统或缓存, 我会查下游调用日志、网关日志和调用耗时。比如日志里显示本服务处理很快, 但调用外部服务耗时 3 秒, 就偏依赖服务问题。

第六步看网络连通和请求耗时。如果怀疑网络, 可以用 `curl -w` 分析 DNS、连接、首包和总耗时, 用 `ping` 或 `telnet/nc` 检查连通性。`netstat` 更适合看连接状态、端口监听和大量 TIME_WAIT / ESTABLISHED 情况。

最后根据证据归因: 如果服务资源高且所有接口都慢, 偏服务性能或资源问题；如果只有某个查询慢且 SQL 扫描大, 偏慢 SQL 或索引问题；如果本服务日志显示下游耗时长, 偏依赖服务；如果连接阶段耗时高或连接异常, 再考虑网络问题。

### 常见追问

- `curl -w` 可以看哪些耗时?
- SQL 慢只在某些参数下出现怎么办?
- 如何判断是下游依赖慢?

### 易错点

- 只看 CPU、内存、磁盘。
- 只手工执行 SQL, 不看慢查询、EXPLAIN 和锁等待。

## 8. EXPLAIN 重点看哪些字段

### 结论

EXPLAIN 重点看访问方式、索引命中、扫描行数、过滤比例和 Extra 信息, 用于识别慢 SQL 风险, 测试侧提供证据给开发评估优化。

### 优秀答案

用 `EXPLAIN` 看 SQL 时, 我会重点关注几个字段。

第一个是 `type`, 它表示表的访问方式, 能大致判断查询效率。一般来说 `const`、`eq_ref`、`ref`、`range` 比较好, `index` 是扫描索引, `ALL` 是全表扫描, 风险最大。如果报表查询接口里大表出现 `ALL`, 就要重点看 where 条件和索引设计。

第二个是 `possible_keys`, 表示 MySQL 认为可能用到的索引；`key` 表示实际使用的索引。如果 `possible_keys` 有值但 `key` 为空, 说明理论上有索引但实际没用上, 可能是条件写法、类型不一致、函数操作字段或索引选择问题。

第三个是 `key_len`, 表示实际使用了索引的长度。它可以帮助判断联合索引用了几个字段。比如联合索引是 `(tenant_id, status, create_time)`, 如果 `key_len` 只用到了第一段, 就要看查询条件是否满足最左前缀原则。

第四个是 `rows`, 表示 MySQL 预估要扫描的行数。`rows` 越大, 查询成本通常越高。如果接口慢, 看到 rows 很大, 就要怀疑筛选条件不够精准或索引不合适。

第五个是 `filtered`, 表示存储引擎返回数据后, server 层过滤后剩余的比例。比例很低时, 说明扫描了很多行但大部分被过滤掉, 也可能需要优化索引。

第六个是 `Extra`, 这里信息很关键。如果出现 `Using filesort`, 说明排序没有很好利用索引；`Using temporary` 说明用了临时表, group by 或 order by 可能有优化空间；`Using index` 表示覆盖索引, 通常是好现象；`Using where` 表示还需要 where 条件过滤。

如果是复杂 SQL, 我还会看 `id` 和 `select_type`, 判断子查询、派生表、联合查询的执行层级。

所以我看 `EXPLAIN` 时, 会综合判断: 有没有走索引、走了哪个索引、扫描行数是否过大、联合索引是否用全、是否出现 filesort / temporary, 再结合接口查询条件和数据量判断慢 SQL 风险。

### 常见追问

- `type = ALL` 说明什么?
- `Using filesort` 一定是问题吗?
- `possible_keys` 有值但 `key` 为空说明什么?

### 易错点

- 只说 type、key、rows。
- 把索引设计说成测试必须负责的优化方案。

## 9. 用户有权限但查不到数据如何定位

### 结论

用户有权限但查不到数据, 要分清权限配置、数据状态、查询参数、接口过滤、SQL 条件、权限缓存和前端展示问题。

### 优秀答案

如果用户反馈“我明明有权限, 但页面上查不到某些数据”, 我会先确认这是权限问题、数据问题, 还是查询链路问题。

第一步确认用户和现象。记录用户账号、角色、部门、租户、查询条件、页面路径、预期能看到的数据 ID 或业务编号, 以及实际查询时间。如果能拿到接口请求参数和响应, 先确认接口返回是空列表、部分缺失, 还是报错但页面没提示。

第二步查接口请求。看前端传给接口的参数是否和用户页面选择一致, 比如时间范围、状态、部门、关键字、分页、排序。很多“查不到”其实是前端默认筛选条件或分页导致的。

第三步查权限数据。用 MySQL 查询该用户的角色、部门、数据范围、租户关系、权限是否启用, 是否存在角色变更后缓存未刷新。还要确认用户是否只拥有菜单权限, 还是也拥有数据权限。有些系统页面能打开, 但数据范围受部门或创建人限制。

第四步查业务数据。按用户说的业务编号或数据 ID 查询主表, 看数据是否存在, 状态是否正常, 是否被逻辑删除, 是否属于用户权限范围内的部门或租户, 是否处于某个不可见状态, 例如草稿、驳回、归档、审批中。

第五步查接口 SQL 口径。如果接口用了多表 join 或权限过滤, 我会根据日志或接口逻辑逐步还原 where 条件, 看是不是 join 条件、状态过滤、部门过滤、租户过滤导致数据被排除。比如用户有部门 A 权限, 但数据实际归属到部门 B, 页面自然查不到。

第六步查 Linux 日志。用请求时间、traceId、用户账号或接口路径查应用日志, 看接口是否有权限过滤日志、SQL 异常、缓存未命中、响应超时或降级。即使页面只显示空数据, 后端日志可能已经有异常或权限拦截信息。

最后根据证据归因。如果数据库里用户确实没有对应数据权限, 就反馈权限配置问题；如果权限正确但 SQL 过滤错, 就提交后端缺陷；如果接口返回有数据但页面不展示, 就偏前端字段映射或渲染问题；如果角色刚变更但缓存未生效, 就定位为缓存刷新或权限同步问题。

### 常见追问

- 菜单权限和数据权限有什么区别?
- 角色刚调整为什么还查不到数据?
- 接口返回有数据但页面没有怎么定位?

### 易错点

- 首先怀疑锁表, 偏离常见原因。
- 不查数据范围、租户、部门、角色和缓存。
