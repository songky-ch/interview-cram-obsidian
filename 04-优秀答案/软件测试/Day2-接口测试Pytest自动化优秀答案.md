## 1. 如何设计一套接口自动化测试用例

### 结论

接口自动化要从业务链路和接口文档两条线入手, 覆盖接口分析、用例设计、数据准备、分层断言、执行报告和回归维护。

### 优秀答案

以天猫精灵生活物联网平台的服务端接口测试为例, 我在设计接口自动化时, 会先从业务链路和接口文档两条线入手。

首先做接口分析。我会梳理接口 URL、请求方法、请求头、鉴权方式、入参、必填字段、响应结构、错误码、是否依赖前置状态, 以及这个接口属于设备接入、物模型、事件上报还是服务调用。对于有上下游依赖的接口, 我会画出接口调用链路, 比如设备创建、绑定、属性上报、服务调用、事件查询这些接口之间的前后置关系。

第二步做用例设计。单接口层面, 我会覆盖正常参数、必填缺失、参数类型错误、边界值、非法枚举、重复提交、无权限 token、过期 token 等场景。业务链路层面, 我会设计接口串联用例, 比如先创建设备或准备设备状态, 再调用物模型服务接口, 最后查询事件或状态是否更新。对于数据分析平台, 则会覆盖创建、提交、审批、驳回、查询、统计报表等核心链路。

第三步准备测试数据。我会区分固定基础数据和动态数据。固定数据用于稳定回归, 动态数据通过前置接口或数据库初始化生成。测试后要清理或隔离数据, 避免污染环境。涉及数据准确性的接口, 我还会用数据库查询或后续查询接口做交叉校验。

第四步设计断言。接口自动化不是只断言状态码 200, 还要断言业务 code、message、关键字段、字段类型、错误码、业务状态变化、数据库落表结果、下游接口查询结果。对于异常用例, 要验证错误提示是否明确, 不能只看接口有没有返回。

第五步用 Pytest 落地。我会按模块组织用例, 把环境地址、token、公共请求方法、测试数据、断言方法拆出来, 用 fixture 管理登录、前置数据和清理动作, 用 parametrize 管理多组参数, 并通过 mark 区分 smoke、regression、接口模块或优先级。

第六步执行和报告。接口自动化可以本地执行, 也可以接入 Jenkins 定时或版本触发执行。失败后我会先区分是脚本问题、环境问题、数据问题还是接口缺陷。确认是缺陷后提交 bug, 附上请求参数、响应、日志、数据库校验结果和复现步骤。

最后是回归维护。接口变更后同步更新用例和断言；历史缺陷用例加入重点回归集；低价值、重复或长期失效的用例定期清理。核心接口和高频缺陷接口优先自动化, 用来保障多版本回归效率和稳定性。

### 常见追问

- 接口自动化如何选择优先级?
- 接口自动化如何设计数据清理?
- 接口失败后如何判断是脚本问题还是接口缺陷?

### 易错点

- 只说接口 URL、入参、状态码, 不讲业务链路和数据校验。
- 只说成功/失败集合, 不分析脚本、环境、数据和接口缺陷原因。

## 2. Pytest 接口自动化框架目录如何组织

### 结论

Pytest 接口自动化框架应按配置、接口封装、公共能力、测试数据、测试用例、报告和执行入口分层。

### 优秀答案

如果让我用 Pytest 设计接口自动化框架, 我会按“配置、接口封装、公共能力、测试数据、测试用例、报告”分层组织。

一个常见目录可以这样设计:

```text
project/
  config/
    config.yaml
    settings.py
  api/
    auth_api.py
    device_api.py
    report_api.py
  common/
    request_client.py
    logger.py
    assert_util.py
    db_helper.py
  data/
    login.yaml
    device_cases.yaml
    report_cases.yaml
  tests/
    conftest.py
    test_device_api.py
    test_report_api.py
  reports/
  pytest.ini
  requirements.txt
  run.py
```

`config` 层主要放环境配置, 比如测试环境地址、数据库连接、超时时间、账号信息的读取方式。`settings.py` 可以封装配置读取逻辑和常量, 避免测试代码里到处写死环境地址。

`api` 层封装业务接口, 比如登录认证、设备接入、物模型、事件查询、报表统计等。每个接口方法只负责组装 URL、请求方法、参数和调用公共请求客户端, 不在这里写测试断言。

`common` 层放通用能力, 比如 `request_client.py` 封装 get/post、header、token、超时、日志打印；`logger.py` 记录请求和响应；`db_helper.py` 做数据库查询校验；`assert_util.py` 封装通用断言, 比如状态码、业务 code、字段存在、字段类型等。

`data` 层放测试数据, 可以用 yaml/json/csv 管理静态参数。对于需要动态生成的数据, 比如设备编号、订单号、报表数据, 我会通过 fixture 或数据工厂生成, 测试结束后清理, 避免污染环境。

`tests` 层只放测试用例。用例文件按 `test_*.py` 命名, 用例函数也按 `test_` 开头。`conftest.py` 用来管理公共 fixture, 比如登录获取 token、准备前置数据、清理测试数据、数据库连接等。用例里尽量只表达测试步骤和断言, 不堆太多底层请求细节。

`reports` 层存放 Allure 或 HTML 测试报告。执行时可以通过 `pytest.ini` 配置默认参数、markers、测试路径、日志级别等, 用 `requirements.txt` 管理依赖, 用 `run.py` 作为统一执行入口。

这样组织的好处是: 用例可读性比较高, 接口变更时只改 api 层或配置层, 公共请求和断言可以复用, 也方便接入 Jenkins 做持续回归。

### 常见追问

- `api` 层和 `common` 层有什么区别?
- `conftest.py` 适合放什么?
- 动态测试数据应该放在哪里?

### 易错点

- 把业务接口封装都放进 common, 导致职责不清。
- 测试用例里写太多底层请求细节。

## 3. 接口链路依赖在 Pytest 中如何处理

### 结论

Pytest 用例之间应尽量独立。单接口能力拆成独立 test, 端到端链路作为场景用例, 前置数据用 fixture 准备。

### 优秀答案

我不会简单地把接口链路全部拆成互相依赖的多个 test, 因为 Pytest 的用例最好保持独立, 不应该依赖执行顺序。否则前一个 test 失败, 后面的 test 会连锁失败, 单独执行某个用例也会不稳定。

我的处理方式会分两层。

第一层是单接口测试。比如登录接口、创建设备接口、绑定设备接口、属性上报接口、查询状态接口, 每个接口都可以单独设计正向、异常、边界、权限等用例。这些用例应该相互独立, 前置数据通过 fixture 或接口准备, 不依赖其他 test 的执行结果。

第二层是业务链路测试。对于“登录 -> 创建设备 -> 绑定设备 -> 上报属性 -> 查询状态”这种端到端链路, 我会设计一个完整的场景用例, 因为它验证的是整条业务链路是否能跑通。这个场景可以写在一个 test 里, 但要通过 Allure step 或日志把每一步拆清楚, 每一步都记录请求、响应和关键断言。这样即使某一步失败, 也能快速定位失败在哪个接口。

同时, 公共前置比如登录 token、测试设备数据、数据库连接, 我会放到 fixture 里管理。fixture 负责准备和清理数据, 场景用例只表达业务步骤和断言。

如果链路很长, 我也可以把每个接口调用封装成 api 层方法或业务 helper, 但测试入口仍然保持清晰: 单接口用例测接口自身能力, 链路用例测跨接口流程, 不让多个 test 之间产生强依赖。

这样既能保证定位清楚, 又能保证自动化用例稳定、可单独执行、适合持续集成回归。

### 常见追问

- 为什么不建议多个 test 互相依赖?
- 端到端链路写成一个 test 会不会难定位?
- fixture 在链路测试中负责什么?

### 易错点

- 认为所有链路步骤都应该拆成多个互相依赖的 test。
- 忽略 Pytest 用例独立性和单独运行稳定性。

## 4. 测试数据污染如何清理和隔离

### 结论

测试数据要分类管理。运行时数据用统一标识和 fixture / yield 清理, 环境隔离要覆盖地址、数据库、账号、租户、数据前缀和生产保护。

### 优秀答案

如果接口自动化会产生设备、报表或业务记录, 我会先对测试数据做分类, 再设计清理和环境隔离策略。

第一类是基础数据, 比如固定账号、固定角色、基础配置、测试租户, 这类数据可以预置在测试环境中, 尽量稳定不频繁改动。

第二类是运行时数据, 比如每次自动化创建的设备、业务单据、报表记录。对于这类数据, 我会在创建时加上统一标识, 比如 `auto_test_时间戳` 或测试批次号, 方便后续查询和清理。

第三类是断言数据, 比如为了验证报表统计或设备状态而生成的数据。这类数据在用例执行期间要保留, 用于接口查询、数据库校验或下游链路验证；用例结束后再清理。

第四类是临时文件或下载结果, 如果只验证下载成功和文件内容, 可以放在临时目录或报告附件中, 执行结束后删除, 不写回静态测试数据目录。

在 Pytest 里, 我一般会用 fixture 管理数据生命周期。比如 fixture 前置创建测试设备或报表数据, `yield` 给用例使用, 用例执行后在 fixture 的后置逻辑里调用删除接口或数据库清理。如果清理失败, 日志里要记录数据 ID, 便于后续人工清理。

环境隔离方面, 我会通过配置区分不同环境的 base_url、数据库连接、账号、租户、设备前缀和数据目录。自动化默认只允许在测试环境或预发布环境执行写操作, 对生产环境写接口要禁用或加保护。不同环境的数据不能混用, 避免测试数据污染正式数据。

另外, 对于历史回归需要长期保留的样本, 我不会直接使用自动化运行中生成的脏数据, 而是沉淀成稳定的静态测试数据或固定初始化脚本。这样既能保证自动化可重复执行, 又能降低环境污染。

### 常见追问

- 清理失败怎么办?
- 生产环境为什么不能直接跑写接口自动化?
- 静态数据和动态数据怎么区分?

### 易错点

- 主要说 teardown, 没有体现 Pytest fixture / yield。
- 把运行时数据写回静态数据目录。
- 只说路径隔离, 不说账号、数据库、租户和生产保护。

## 5. 查询类接口如何设计断言

### 结论

查询类接口断言要分层: 基础响应、JSON 结构、业务数据、分页排序、数据库校验、权限范围和异常返回。

### 优秀答案

以数据分析平台的报表列表查询接口为例, 我不会只断言 HTTP 状态码 200, 而是会分层设计断言。

第一层是基础响应断言。比如 HTTP 状态码是否正确, 业务 code 是否为成功, message 是否符合预期, 响应时间是否在可接受范围内。

第二层是 JSON 结构断言。检查响应里是否包含 `total`、`list`、`pageSize`、`pageNum` 等字段, `list` 是否为数组, 每条报表数据里的 id、名称、状态、创建时间、统计值等字段是否存在, 字段类型是否正确, 必填字段是否为空。

第三层是业务数据断言。比如我按状态筛选“已完成”, 返回列表里的所有记录状态都必须是已完成；按时间范围查询, 返回数据的创建时间必须都在范围内；按部门或用户查询, 返回数据不能超出当前账号的数据权限。

第四层是分页和排序断言。比如 pageSize 为 10 时, 当前页数据不能超过 10 条；total 要和数据库或查询接口统计结果一致；按创建时间倒序时, 列表时间顺序要正确。

第五层是数据库或下游校验。如果这个报表数据要求准确, 我会用 SQL 按相同筛选条件查数据库, 对比数量、状态、统计值、时间范围是否一致。不能只看接口返回成功。

第六层是权限断言。不同角色调用同一个报表列表接口时, 管理员、普通用户、只读用户看到的数据范围和可操作字段应该不同。无权限 token 或越权参数应该返回明确错误, 不能返回其他部门数据。

第七层是异常断言。比如缺少必填参数、非法页码、非法状态、超大 pageSize、过期 token, 要断言错误码、错误提示和返回结构符合接口约定。

所以查询类接口的断言重点不是“接口有没有返回”, 而是返回数据是否符合筛选条件、权限范围、分页排序、数据库统计口径和业务预期。

### 常见追问

- 报表 total 如何验证?
- 权限断言怎么设计?
- 查询接口异常场景有哪些?

### 易错点

- 只讲状态码、包含断言、不等断言等语法。
- 没有落到业务字段、分页排序、权限范围和数据库校验。

## 6. Jenkins 偶现失败如何定位

### 结论

CI 偶现失败要先保留失败现场, 再从环境、数据、执行方式、脚本和服务端日志逐层定位。

### 优秀答案

如果 Jenkins 上偶尔失败, 本地重跑又通过, 我会先把它当成不稳定问题来定位, 不会直接判定是脚本问题或接口缺陷。

第一步看失败现场。先查看 Jenkins 构建号、执行时间、失败用例、请求 URL、请求参数、响应体、错误码、traceId、测试数据 ID 和 Allure 报告。重点确认失败是断言失败、请求超时、连接失败、数据为空, 还是返回了业务错误。

第二步对比环境。检查 Jenkins 和本地是否使用同一个 base_url、账号、token、数据库、配置文件、环境变量、Python 版本和依赖版本。很多问题是 Jenkins 指向了不同环境或使用了过期配置。

第三步排查测试数据。看 Jenkins 执行时用的数据是否被其他任务污染, 是否存在重复数据、脏数据、状态不满足前置条件、数据被提前清理。报表接口尤其要关注数据时效性, 比如统计任务还没跑完, Jenkins 就开始断言 total 或列表结果。

第四步看执行方式。Jenkins 可能定时执行、并行执行或多个任务共享同一环境, 导致接口限流、数据冲突、执行顺序变化。本地串行重跑通过, 不代表 CI 场景没有问题。所以我要确认是否存在用例间依赖、共享账号、共享数据、并发清理等问题。

第五步结合服务端日志。如果接口返回 500、超时或业务错误, 我会拿 traceId、请求时间和参数去查服务端日志、网关日志、数据库日志, 看是服务异常、慢查询、缓存问题还是接口本身偶现缺陷。

第六步做复现和隔离。可以在 Jenkins 上单独重跑该用例、多次循环执行、固定同一组数据执行, 或把用例改成独立数据后再跑。如果独立数据后稳定通过, 多半是数据污染或用例依赖；如果固定请求仍然偶现失败, 就更像接口稳定性问题。

最后根据证据分类处理: 配置或依赖不一致就修 CI 环境；数据污染就改 fixture 和清理策略；脚本断言不稳定就优化断言；服务端偶现错误就提交缺陷, 附上请求、响应、traceId、日志时间和复现概率。

### 常见追问

- Jenkins 上为什么不能主要靠打断点?
- 本地通过但 CI 失败可能有哪些原因?
- 偶现失败如何证明是服务端问题?

### 易错点

- 只比较本地和 Jenkins 配置, 不看并发、数据污染和服务端日志。
- 没有保留 traceId、数据 ID、构建号等失败现场。

## 7. 发布前时间不足如何选择接口回归范围

### 结论

时间不足时按风险优先级选择回归集, 优先执行核心冒烟、高风险变更、历史缺陷和本次改动影响链路, 并明确阻塞发布标准和回滚方案。

### 优秀答案

如果 300 条接口自动化用例全量执行要 40 分钟, 但发布前只剩 10 分钟, 我不会盲目全量跑, 会按风险优先级选择回归范围。

首先, 平时用例维护时我会给用例打标签, 比如 `smoke`、`core`、`regression`、`high_risk`、`history_bug`。发布前 10 分钟优先执行核心冒烟集和高风险回归集, 包括核心业务链路、登录鉴权、关键查询、核心写入接口、权限控制、历史严重缺陷相关用例。本次变更影响到哪些接口, 这些接口及上下游链路也必须优先跑。

其次, 我会结合本次改动范围选择用例。如果只改了报表统计接口, 就重点跑报表查询、筛选、分页、权限、数据库统计口径和相关历史缺陷；如果改了公共鉴权或网关层, 影响面大, 就要扩大到多个模块的核心接口。

第三, 我会给发布结论设门槛。核心链路失败、鉴权失败、数据错误、权限越权、核心接口 500 或超时, 这些属于阻塞发布。非核心字段提示文案、低风险展示问题, 可以记录风险后由产品和项目负责人确认是否放行。

第四, 10 分钟内跑完精选集后, 不能就结束。全量回归可以继续异步执行, 同时发布后做线上只读验证、日志监控、错误率监控和核心接口探测。如果全量回归或线上监控发现高风险问题, 要根据影响范围决定是否暂停发布、修复补丁或执行回滚。

最后, 我会提前确认回滚方案, 包括版本包、配置、数据库变更是否可回滚、负责人是谁、触发条件是什么。这样在时间不足时, 不是靠运气发布, 而是用风险分级、核心用例、变更影响分析和回滚预案控制风险。

### 常见追问

- 哪些接口失败必须阻塞发布?
- 如何定义核心冒烟集?
- 回滚方案需要确认哪些内容?

### 易错点

- 把高风险问题留到发布后才发现。
- 把“一键回滚”说得过于简单, 忽略数据库和配置变更。

## 8. 如何用 pytest.mark 管理接口自动化用例

### 结论

用 pytest.mark 按发布阶段、优先级、模块、历史缺陷和高风险变更管理用例, 并和 CI 执行策略绑定。

### 优秀答案

在 Pytest 接口自动化里, 我会用 `pytest.mark` 对用例做分组管理, 方便按发布阶段、业务模块、优先级和风险范围选择执行。

首先会在 `pytest.ini` 里注册自定义 marker, 避免执行时出现未注册警告。例如:

```ini
[pytest]
markers =
    smoke: core smoke cases for release validation
    regression: full regression cases
    P0: blocker or critical cases
    P1: high priority cases
    P2: medium or low priority cases
    auth: authentication module
    report: report module
    device: device module
    history_bug: historical defect regression cases
    high_risk: high risk change related cases
```

然后在用例上打标签。单个接口用例可以按函数打标签, 一个模块下的所有用例可以按类打标签。例如报表查询核心用例可以同时打 `smoke`、`P0`、`report`:

```python
@pytest.mark.smoke
@pytest.mark.P0
@pytest.mark.report
def test_report_list_query_success():
    ...
```

如果某个类都是设备模块用例, 可以在类上打模块标签:

```python
@pytest.mark.device
class TestDeviceApi:
    ...
```

执行时可以按标签组合选择。例如发布前只跑核心冒烟:

```bash
pytest -m "smoke and P0"
```

只跑报表模块回归:

```bash
pytest -m "report and regression"
```

只跑高优先级或历史缺陷:

```bash
pytest -m "P0 or history_bug"
```

在 CI 里, 我会把标签和流水线策略绑定。比如代码提交后跑 `smoke and P0`, 发布前跑 `smoke or high_risk or history_bug`, 夜间定时跑全量 `regression`。

同时 marker 需要维护, 不能所有用例都标成 smoke。冒烟集应该只保留核心链路和阻塞发布的用例；历史缺陷修复后要加入 `history_bug`；低价值或重复用例要定期清理, 保证回归效率。

### 常见追问

- marker 为什么要注册?
- 冒烟集如何防止越来越大?
- CI 不同阶段分别跑哪些 marker?

### 易错点

- 只会 `pytest -m smoke`, 不会组合标签。
- marker 没有和发布风险、CI 策略绑定。

## 9. 如何用参数化和 YAML 管理测试数据

### 结论

稳定查询条件适合放 YAML 并用 pytest.mark.parametrize 参数化, 动态数据和敏感信息应由 fixture、环境变量或配置管理。

### 优秀答案

如果同一个报表查询接口要覆盖不同状态、时间范围和角色权限, 我会用 Pytest 参数化加外部测试数据来实现。

对于比较稳定的查询条件和预期结果, 我会放到 YAML 文件里, 比如:

```yaml
test_report_search:
  - case_name: "管理员-查询全部状态-今日报表"
    role: "admin"
    payload:
      status: "ALL"
      time_range: "TODAY"
      page: 1
      page_size: 10
    expected:
      status_code: 200
      biz_code: 0
      fields: ["total", "list"]
      allow_empty: true

  - case_name: "普通用户-查询已完成-本周报表"
    role: "user"
    payload:
      status: "DONE"
      time_range: "THIS_WEEK"
      page: 1
      page_size: 10
    expected:
      status_code: 200
      biz_code: 0
      status_should_be: "DONE"
      only_current_user_scope: true
```

测试代码里用工具函数读取 YAML, 再通过 `pytest.mark.parametrize` 参数化:

```python
cases = load_yaml("data/report_cases.yaml")["test_report_search"]

@pytest.mark.parametrize("case", cases, ids=[case["case_name"] for case in cases])
def test_report_search(case, token_by_role):
    token = token_by_role(case["role"])
    resp = report_api.search(case["payload"], token=token)
    assert resp.status_code == case["expected"]["status_code"]
```

测试数据放外部文件的好处是数据和代码分离, 用例结构更清晰, 后续新增状态、角色、时间范围时不用改测试逻辑, 只维护数据文件。也方便用例评审, 产品或测试同事能直接看懂覆盖了哪些场景。

但我不会把所有东西都放 YAML。比如 token、动态报表数据、测试后清理, 更适合用 fixture 生成和管理。敏感信息如账号密码也不应写在 YAML 里, 应通过环境变量或配置文件读取。

另外, 参数组合要控制规模。状态、时间范围、角色如果全量笛卡尔积会导致用例爆炸, 我会按风险选择代表性组合: 核心角色、核心状态、边界时间、无权限场景、历史缺陷场景优先覆盖。

### 常见追问

- 为什么不把所有测试数据都写在代码里?
- 为什么不能把所有组合都做笛卡尔积?
- token 和动态数据怎么处理?

### 易错点

- 把外部文件的主要价值说成权限隔离。
- 把敏感账号密码放进 YAML。
- 参数组合无限膨胀。

## 10. 接口返回字段经常变动时如何维护断言

### 结论

核心字段强断言, 接口契约用 JSON Schema 校验, 非核心字段弱断言, 删除或类型变化必须做影响分析和关联接口回归。

### 优秀答案

如果接口返回字段经常变动, 我不会把整个响应 JSON 做完全等值断言, 因为这样用例会非常脆弱。但也不能完全不校验字段, 否则真实问题发现不了。

我的做法是分层维护断言。

第一层是核心业务字段强断言。比如报表查询接口里的 `id`、`status`、`total`、`list`、创建时间、统计值、权限范围这些核心字段, 只要业务依赖它, 就必须断言字段存在、类型正确、取值符合预期。核心字段不能随意删除或改变类型。

第二层是接口契约断言。对响应结构、必填字段、字段类型、枚举值范围, 可以用 JSON Schema 做校验。这样不要求整个 JSON 完全一致, 但能发现字段类型从数字变字符串、必填字段缺失、结构层级变化这类问题。

第三层是非核心字段弱断言。比如展示辅助字段、描述字段、扩展字段, 不做固定值等值断言, 只校验类型、是否可为空、长度范围或格式。后端新增字段时, 旧用例不应该因为多了字段就失败。

第四层是变更影响分析。如果是新增字段, 我会确认是否影响前端展示、下游接口、数据库落表和权限控制, 需要的话补充对应断言。如果是删除字段、字段类型调整、枚举值变化, 我会先确认接口文档和调用方是否同步变更, 因为这属于可能破坏兼容性的变更。

第五层是关联接口回归。比如新增字段会影响查询、导出、统计或详情接口, 就要同步检查这些关联接口的断言和期望值, 防止只改了一个接口用例, 但下游链路漏测。

所以我的原则是: 核心字段严格断言, 非核心字段结构化校验, 新增字段不让用例过度敏感, 删除和类型变化必须触发契约和影响范围评估。

### 常见追问

- 新增字段是否应该导致旧用例失败?
- 删除字段为什么风险更高?
- JSON Schema 适合解决什么问题?

### 易错点

- 对整个 JSON 做完全等值断言, 导致用例过度脆弱。
- 只追求用例不失败, 忽略核心字段和契约变更。
