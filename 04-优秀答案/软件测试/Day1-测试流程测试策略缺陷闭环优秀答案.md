## 1. AI Agent 新需求如何做测试分析和测试策略设计

### 结论

AI Agent 新需求不能只按传统页面功能拆测试点, 要从业务目标、任务链路、知识库、Prompt、工具调用和最终输出质量出发设计测试策略。

### 优秀答案

在睿思 AI Agent 项目里, 我接到新需求后, 不会只按传统 Web 页面功能去拆, 而是先从业务目标和 Agent 的任务链路出发做测试分析。

首先我会参加需求评审, 明确这个 Agent 要解决什么现网维护问题, 例如现网日志分析、分析件初筛、业务问答或者多轮对话处理。然后我会拆测试范围: 输入是什么, Agent 是否需要调用工具, 是否依赖知识库或 Prompt, 中间分析步骤是否符合业务规则, 最终输出结论是否准确、完整、可解释。

测试策略上, 我会重点从几类场景覆盖。第一类是正常业务场景, 用真实或脱敏的现网日志、高频问题、历史分析样本验证 Agent 是否能给出正确结论。第二类是异常和边界场景, 比如超长日志、无效输入、缺字段数据、模糊指令、重复提问, 验证 Agent 是否能正确拒答、提示原因或引导用户补充信息。第三类是多轮对话和上下文场景, 验证上下文继承是否正确, 是否会遗忘前置信息或错误引用历史内容。第四类是工具调用场景, 重点检查触发条件、入参传递、执行结果解析、异常返回和权限边界。

质量风险方面, 我最关注几个点: 一是回答准确性, 防止 Agent 在没有依据的情况下产生幻觉；二是业务一致性, 同一类任务多次执行结果不能明显偏离业务事实；三是工具调用风险, 不能误触发、错传参或越权操作；四是异常处理能力, 当日志不完整、工具失败或知识库无答案时, 要能明确提示而不是胡乱输出；五是回归风险, Prompt、知识库或工作流变更后, 要用历史缺陷和典型样本做回归评测。

最后我会把测试结果和验收标准对齐, 例如核心场景通过率、任务完成率、工具调用成功率、历史缺陷回归结果、严重缺陷数量等, 通过缺陷闭环和上线验收保证版本质量。

### 常见追问

- AI Agent 回归测试数据集如何设计?
- Agent 输出不固定时, 如何判断通过或失败?
- Prompt 变更后, 是否需要完整回归?

### 易错点

- 只讲传统功能测试, 不讲 Agent 的准确性、幻觉、上下文和工具调用。
- 没有结合现网日志分析、分析件初筛、业务问答等项目场景。
- 只说测试点, 不说验收标准。

## 2. AI Agent 回归测试数据集如何设计

### 结论

AI Agent 回归数据集不只包含输入样本, 还要包含标准答案、关键要点、禁止项、工具调用预期和评估规则。

### 优秀答案

在睿思 AI Agent 项目里, 我设计回归测试数据集时, 不会只保存输入样本, 还会同时维护预期结果和评估规则。因为 Agent 输出不是简单字段比对, 必须明确什么算通过, 什么算偏离业务事实。

数据集我一般会分几类。

第一类是核心正向样本, 例如真实或脱敏后的现网日志、分析件初筛样本、高频业务问答, 这些样本要有标准结论、关键判断依据和期望输出要点, 用来验证 Agent 的核心任务完成率和回答准确性。

第二类是异常和负向样本, 比如无效输入、日志缺字段、格式错误、无关问题、知识库没有答案的问题。重点验证 Agent 是否能明确说明无法处理的原因, 而不是编造答案。

第三类是边界样本, 包括超长日志、大文件、极短输入、多个故障信息混在一起、相似问题不同结论等。这里主要验证 Agent 的鲁棒性、稳定性、上下文处理和输出一致性。

第四类是多轮对话样本, 例如先给一段日志, 再追问原因、处理建议、是否需要升级, 用来验证上下文继承、意图识别和前后回答一致性。

第五类是工具调用样本, 包括应该触发工具、不应该触发工具、工具返回异常、入参缺失等场景, 验证工具触发条件、参数传递、异常处理和权限边界。

第六类是历史缺陷样本, 把之前发现过的幻觉、误判、错调用、上下文丢失、异常不提示等问题沉淀下来, 每次 Prompt、知识库或工作流变更后都跑一遍, 防止回归。

最终这套数据集会按场景、风险等级和优先级维护, 回归时关注核心样本通过率、历史缺陷是否复现、回答是否命中关键要点、工具调用是否符合预期, 这样才能支撑版本迭代中的效果对比和上线验收。

### 常见追问

- 数据集中的标准答案如何维护?
- 历史缺陷样本如何进入回归集?
- 超长日志样本主要验证什么?

### 易错点

- 只说正向、负向、边界, 不说期望结果和评估规则。
- 只按等价类边界值回答, 缺少多轮对话、工具调用和幻觉样本。

## 3. Agent 输出不固定时如何判断用例通过或失败

### 结论

Agent 输出不能做完全字符串匹配, 要按业务结论、关键事实、依据可追溯、禁止项和工具调用预期判断。

### 优秀答案

AI Agent 的回答本身有一定生成随机性, 所以我不会用接口测试那种完全字符串一致来判断通过失败, 而是按“语义和业务结果是否正确”来判断。

在睿思 AI Agent 项目里, 我会先给每条用例定义评估标准, 包括几个方面: 第一, 最终结论是否符合业务预期；第二, 回答里必须包含的关键要点是否命中；第三, 是否引用了真实日志、知识库或工具返回结果作为依据；第四, 是否出现了禁止项, 比如编造不存在的告警、错误推断原因、给出越权操作建议；第五, 如果涉及工具调用, 触发条件、入参和结果解释是否正确。

对于表达不同但结论一致的情况, 只要关键事实正确、业务判断正确、没有幻觉、没有误导性建议, 我会判定通过。比如同一个现网日志分析任务, Agent 可能用不同表述说明故障原因, 但只要故障类型、定位依据和处理建议一致, 就可以通过。

如果最终结论看似正确, 但中间过程引用了不存在的数据、编造了日志信息, 或者给出了不符合业务规则的推理过程, 我会判定失败。因为这类问题在线上会降低用户信任, 也可能导致维护人员按错误依据处理。

落地执行上, 我会结合规则校验和人工评审。核心回归样本可以维护标准答案、关键要点和禁止项；对高风险样本多跑几次, 看输出是否稳定；再和历史版本结果做对比, 判断 Prompt、知识库或工作流变更后有没有效果退化。

### 常见追问

- 如果结论正确但推理过程有幻觉, 是否通过?
- 多次执行结果略有差异, 怎么处理?
- 人工评审和规则校验如何结合?

### 易错点

- 用完全字符串一致判断 Agent 输出。
- 只看最终结论, 忽略过程中的幻觉和错误依据。

## 4. 只改 Prompt 时是否需要完整回归

### 结论

Prompt 改动必须做回归, 但回归范围应按影响面分层。核心链路、历史缺陷和高风险样本必须回归, 低风险场景可以抽样。

### 优秀答案

如果版本只改了 Prompt, 我不会因为没改代码就降低风险。AI Agent 里 Prompt 会直接影响意图识别、工具调用、输出格式和业务结论, 所以一定要做回归评估。

我会先做 Prompt diff 分析, 看它改的是目标、上下文、约束、输出格式, 还是工具调用说明。不同改动对应不同测试范围。

如果改的是任务目标或业务判断逻辑, 影响面最大, 我会覆盖核心业务场景、历史缺陷样本、多轮对话样本和工具调用样本, 接近完整回归。比如现网日志分析的判断目标发生变化, 就必须验证最终结论、分析依据和处理建议是否符合新目标。

如果改的是上下文或业务背景, 我会重点确认新上下文是否和现网业务事实一致, 同时更新受影响的测试样本和标准答案, 防止用旧预期判断新逻辑。

如果改的是约束条件, 比如要求不能越权、不能输出某类操作建议、必须提示用户确认, 我会重点覆盖边界场景、禁止项、异常输入和越权风险场景。

如果改的是输出格式, 我会重点验证结构化输出、字段完整性、可读性, 以及下游是否依赖这些字段。

所以我的策略不是简单地“一律全量”或“只冒烟”, 而是按影响面分层回归: 核心场景、历史缺陷、高风险样本必须跑；受影响模块重点跑；低风险场景可以抽样。最后通过回答准确性、工具调用正确性、历史缺陷是否复现和输出稳定性来判断是否达标。

### 常见追问

- Prompt 只改了一句话, 为什么还要回归?
- 如何判断 Prompt 变更影响范围?
- 输出格式变更会影响哪些测试?

### 易错点

- 直接说一律完整回归, 没有成本控制意识。
- 直接说只做冒烟, 低估 Prompt 对结果质量的影响。

## 5. 平台复杂需求如何设计测试方案

### 结论

平台复杂需求要先拆端到端业务链路, 再分层覆盖功能、权限、接口、数据库、报表统计、多模块联动和自动化回归。

### 优秀答案

如果一个平台需求同时涉及数据流转、权限控制、报表展示和多模块联动, 我会先把它拆成一条端到端业务链路, 再按风险点设计测试方案。

以生产维护数据分析平台为例, 我会先明确数据从哪里产生, 经过哪些状态, 哪些角色可以操作, 最终在哪些报表或页面展示。比如一条数据从创建、提交、审核、驳回、重新提交、处理完成, 到最后进入统计报表, 每一步都要确认状态变化、操作权限和数据结果。

功能测试方面, 我会覆盖正常流转、驳回、重复提交、撤回、重新提交、状态变更后页面展示等场景, 确保流程完整。对于多模块联动, 我会重点验证上游操作后, 下游列表、详情、统计报表、消息或任务状态是否同步更新。

权限测试方面, 我会按角色设计用例, 比如普通用户、审核人、管理员、只读角色。验证有权限的人能正确操作, 无权限的人不能操作, 不能通过改接口参数或直接访问 URL 越权查看、修改其他部门或其他用户的数据。

数据校验方面, 我会同时看前端展示、接口返回和数据库记录是否一致。比如报表里的数量、状态、时间范围、筛选条件、统计口径是否和实际数据一致, 避免只看页面显示正确但底层数据错误。

接口测试方面, 我会把核心接口提取出来, 包括创建、提交、审批、驳回、查询、统计等接口, 用 Pytest 设计接口自动化用例。重点覆盖正常参数、异常参数、权限 token、状态前置条件和多接口串联场景, 用来支撑后续回归。

最后验收时, 我会关注几个标准: 核心流程是否全链路通过, 权限是否无越权, 报表数据是否准确, 接口自动化是否覆盖核心链路, 严重缺陷是否全部关闭, 历史缺陷是否回归通过。

### 常见追问

- 多模块联动怎么设计用例?
- 权限测试如何防止只测页面不测接口?
- 报表数据如何校验准确性?

### 易错点

- 只列功能点, 不讲端到端业务链路。
- 只看页面展示, 不校验接口和数据库。

## 6. 页面报表和接口返回不一致如何定位

### 结论

先固定不一致现象, 再按请求参数、接口响应、前端渲染、分页聚合、数据库、权限范围、缓存或异步同步逐层定位。

### 优秀答案

如果发现页面报表展示数据和接口返回数据不一致, 我会先确认不一致的具体表现, 比如是数量不一致、字段值不一致、排序不一致, 还是筛选条件下的数据范围不一致。先把问题现象固定下来, 避免泛泛定位。

第一步我会看前端请求参数是否正确, 包括时间范围、部门、状态、分页参数、排序字段、筛选条件等。很多报表问题不是接口返回错, 而是前端传参和页面选择条件不一致。

第二步对比接口响应和页面渲染。看接口字段名、字段类型、枚举值、单位、精度、空值处理是否和前端约定一致。比如后端返回的是状态码, 前端映射成中文状态时映射表错了；或者金额、百分比、时间格式在前端处理时出现偏差。

第三步确认分页和聚合逻辑。报表有时接口返回的是分页明细, 页面展示的是汇总值；或者页面只渲染当前页, 但统计值应该来自全量数据, 这时要确认前后端统计口径是否一致。

第四步查数据库或日志, 看接口返回的数据本身是否正确。可以用 SQL 按同样筛选条件查数据库, 核对数据条数、状态、时间范围和统计口径。如果数据库和接口一致, 更偏向前端展示问题；如果数据库和接口不一致, 就要查后端查询逻辑、数据同步或缓存。

第五步考虑权限和数据隔离。不同角色看到的数据范围可能不同, 要确认当前登录用户的角色、部门权限和接口 token 是否一致。

第六步看缓存或异步同步。报表类数据可能有缓存、定时任务或离线统计, 页面和接口访问的不是同一份实时数据, 需要确认刷新时机和数据更新链路。

最终我会用证据定位归属: 如果请求参数正确、接口返回正确但页面错, 就提交前端渲染或字段映射问题；如果接口和数据库不一致, 就提交后端查询或统计逻辑问题；如果数据库源数据就错, 就继续追数据流转或同步链路的问题。

### 常见追问

- 如何判断是前端问题还是后端问题?
- 分页统计不一致怎么定位?
- 缓存导致数据不一致怎么确认?

### 易错点

- 只怀疑前端渲染, 不查接口请求参数和数据库。
- 忽略权限范围、缓存、异步同步和统计口径。

## 7. MySQL 如何验证统计口径问题

### 结论

先统一页面和接口的筛选条件, 再按 count、group by、sum / avg、join、distinct、权限范围和数据时效性拆解验证。

### 优秀答案

如果怀疑是 MySQL 统计口径问题, 我会先把页面和接口的查询条件固定下来, 包括时间范围、部门、状态、用户角色、分页、排序和其他筛选条件, 确认测试时使用的是同一组参数。

然后我会根据报表口径拆 SQL 验证。

如果是数量不一致, 我会先用 count(*) 按相同条件查总数, 再确认是否需要排除软删除数据、无效状态数据、草稿数据或测试数据。很多统计问题不是 SQL 写错, 而是状态范围没对齐。

如果是分类统计不一致, 我会按状态、类型、部门或日期做 group by, 看每一类的数量是否和报表一致。这样可以快速定位是哪一类数据偏差。

如果是金额、耗时、成功率这类指标, 我会用 sum、avg、count 分别验证分子和分母, 不只看最终百分比。比如成功率异常, 要分别查成功数量和总数量, 看是过滤条件错了还是状态判断错了。

如果涉及多表关联, 我会检查关联条件是否正确, 有没有一对多导致重复统计, 是否需要 distinct, 以及关联字段是否存在脏数据或空值。

如果涉及权限范围, 我会确认当前用户能看到哪些部门或数据范围, 再把权限条件加到 SQL 中验证, 防止页面统计的是当前角色数据, 而我手工 SQL 查的是全量数据。

最后还要确认数据时效性, 比如报表是否来自实时表、汇总表、缓存表或定时任务。如果页面读取的是汇总表, 而我查的是明细表, 两边可能因为同步延迟出现差异。

所以我的验证思路是: 先统一筛选条件, 再按报表口径拆成 count、group by、sum / avg、join、distinct、权限范围和数据时效性逐项验证, 最后用 SQL 结果和接口、页面三方对齐来定位问题。

### 常见追问

- count 结果不一致可能有哪些原因?
- 一对多关联导致重复统计怎么发现?
- 成功率异常怎么拆分验证?

### 易错点

- 只写 select count(*)。
- 忽略 group by、去重、关联表、权限范围和同步延迟。

## 8. 严重缺陷如何推动生命周期闭环

### 结论

严重缺陷要先固定现场和影响范围, 再提交高质量缺陷单, 跟进修复方案、关联回归、上线验证和回归资产沉淀。

### 优秀答案

如果我发现一个严重缺陷, 我会先固定现场, 再推动修复闭环。

第一步是确认和记录问题。比如在数据分析平台里发现报表统计结果错误, 我会记录环境、版本、账号角色、操作路径、筛选条件、接口 URL、请求参数、响应结果、页面截图、数据库校验结果和预期结果。如果是 AI Agent 项目, 我还会记录输入样本、完整对话上下文、工具调用日志、Agent 输出、预期业务结论以及是否存在幻觉或误判。

第二步是判断严重程度和影响范围。我会确认这个缺陷是否影响核心业务链路, 是否会导致数据错误、权限越权、任务失败或误导用户。如果属于阻塞上线或高风险问题, 我会标记为高优先级, 同步研发、产品和测试负责人, 说明影响范围和复现证据。

第三步是提交 bug。缺陷单里我会写清楚标题、复现步骤、实际结果、预期结果、附件证据、日志或接口信息、影响范围、严重等级和建议排查方向。这样开发能快速定位, 减少来回沟通成本。

第四步是跟进修复过程。开发定位后, 我会关注修复方案是否只修当前现象, 还是覆盖了根因。如果修复涉及接口、数据库、权限、Prompt、知识库或工作流, 我会同步评估回归范围。

第五步是测试环境验证。修复后我先按原路径复现验证, 确认缺陷不再出现；然后验证关联功能, 比如上游数据流转、下游报表、接口返回、权限控制、历史缺陷样本等, 防止修复引入新问题。

第六步是回归和上线验收。严重缺陷修复后, 我会跑核心流程和相关自动化用例, 输出回归结论。如果允许在生产验证, 上线后我会用不影响生产数据的方式做线上验证, 或通过日志、监控、只读查询确认问题已修复。

最后我会把这个缺陷沉淀为回归用例或 AI Agent 历史缺陷样本, 后续版本持续回归, 避免同类问题再次出现。

### 常见追问

- 偶现缺陷怎么记录?
- 严重缺陷修复后怎么确定回归范围?
- 上线后如何验证不影响生产数据?

### 易错点

- 只说提交给开发, 不讲影响范围和推动闭环。
- 只复测原问题, 不回归关联链路。

## 9. 开发认为不是 bug 时如何处理

### 结论

不要争论结论, 要回到需求、验收标准和用户影响, 用证据推动产品、开发、测试共同做风险决策。

### 优秀答案

如果开发说这不是 bug, 是需求设计, 我不会先和开发争论结论, 而是先回到需求和用户影响上确认。

第一步我会拿需求文档、原型、接口约定或验收标准对齐, 看当前行为是否符合已确认的设计。如果文档明确写了这个逻辑, 那它可能不是代码缺陷；如果文档没写清楚或和实际行为不一致, 就需要产品补充确认。

第二步我会补充测试证据, 说明它为什么可能影响用户或上线质量。比如这个行为会导致数据统计误解、权限边界不清、流程中断、Agent 给出误导性结论, 或者会影响核心用户场景。我会尽量用具体场景和复现路径说明风险, 而不是只说“我觉得不合理”。

第三步组织产品、开发、测试一起确认结论。如果确认是缺陷, 就按缺陷流程修复；如果确认是需求设计, 我会推动产品明确验收标准, 并把风险记录到测试报告或上线风险项里, 包括影响范围、触发条件、用户影响和建议处理方案。

第四步如果风险较高, 我会建议上线前采取措施, 比如调整提示文案、限制入口、增加二次确认、补充运营说明, 或将优化项排入后续版本。对于阻塞级风险, 我会明确给出测试不建议上线的结论, 由产品或项目负责人做最终决策。

第五步上线后跟踪反馈和日志。如果这个点后续确实引发用户问题, 可以快速回溯到风险记录和测试证据, 推动后续修复。

这样处理的核心不是证明谁对谁错, 而是让风险透明, 让上线决策有依据。

### 常见追问

- 产品也认为不用改怎么办?
- 什么情况下测试会给出不建议上线?
- 风险记录应该包含什么?

### 易错点

- 说“让用户适应”, 会显得质量推动不足。
- 只记录风险, 不推动验收标准和上线决策。

## 10. 如何完整讲述一次项目测试流程

### 结论

项目化回答要按项目背景、需求分析、测试方案、测试执行、缺陷闭环、回归验收、结果沉淀组织, 并绑定真实项目场景。

### 优秀答案

以睿思 AI Agent 的现网日志分析需求为例, 我拿到需求后, 首先会参加需求评审, 明确这次需求要解决什么业务问题, 比如是新增日志分析能力、分析件初筛能力, 还是优化业务问答效果。

需求分析阶段, 我会先判断这个需求是否和历史能力有耦合。如果只是新增一个独立问答场景, 可以单独设计测试点；如果涉及 Prompt、知识库、工具调用或已有业务流程, 就要分析它对历史场景的影响, 特别是是否会影响原来的高频问答、日志分析结论和工具调用链路。

测试方案阶段, 我会把测试范围拆成几层。第一层是核心功能, 验证 Agent 能否根据现网日志给出正确结论和处理建议。第二层是异常场景, 比如无效输入、超长日志、缺字段日志、模糊描述。第三层是多轮对话, 验证上下文是否能正确继承。第四层是工具调用, 验证触发条件、入参、返回结果解析和异常处理。第五层是回归样本, 用历史缺陷、高频问题和典型日志做效果对比。

测试执行阶段, 我会按优先级先验证核心链路, 再覆盖异常和边界场景。执行过程中如果发现问题, 我会记录输入样本、完整对话、Agent 输出、预期结论、工具调用日志、接口响应和复现步骤。如果是偶现问题, 会多次执行确认出现条件和稳定性。

缺陷跟踪阶段, 我会根据影响范围判断严重等级。比如 Agent 编造日志信息、误判故障原因、错误触发工具或越权操作, 都属于高风险问题。开发或算法同学修复后, 我不仅复测原缺陷, 还会回归关联场景, 防止 Prompt 或工作流调整后影响其他能力。

上线验收阶段, 我会重点看核心样本通过率、历史缺陷是否复现、工具调用是否符合预期、异常提示是否合理、严重缺陷是否关闭。通过后再输出测试结论和上线风险说明。

最后, 我会把本次需求中的典型样本、历史缺陷、优秀测试点和回归数据沉淀到测试知识库, 后续版本继续作为回归评测数据使用。

### 常见追问

- 如何判断需求是否和历史能力耦合?
- 测试方案里如何体现优先级?
- 上线验收标准有哪些?

### 易错点

- 只讲通用测试流程, 不绑定项目。
- 没有讲缺陷闭环和测试资产沉淀。
