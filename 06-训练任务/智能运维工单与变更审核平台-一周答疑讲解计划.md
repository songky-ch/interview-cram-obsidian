# 智能运维工单与变更审核平台一周答疑讲解计划

## 计划目标

围绕“智能运维工单与变更审核平台”这个模拟项目, 用一周时间把简历中的测试技术栈串成可讲、可问、可练的项目经验。重点不是背工具, 而是能说清楚每项技术在项目哪个阶段使用、解决什么问题、怎么实践、面试官追问时怎么展开。

## 项目主线

平台接入设备告警、现网日志、巡检数据和用户报障信息, 由 AI Agent 完成故障初筛、日志分析、知识库问答和处理建议生成。运维人员确认后创建工单, 发起变更申请, 按风险等级进入多级审核, 审批通过后执行变更或回滚, 最后验收闭环并沉淀知识库。

核心流程:

告警/日志接入 -> AI Agent 初筛 -> 工单创建 -> 变更申请 -> 多级审核 -> 执行/回滚 -> 验收关闭 -> 知识库沉淀

## 涉及技术栈与项目应用

| 技术栈 | 项目阶段 | 实际应用 |
|---|---|---|
| 软件测试流程 | 需求评审、测试计划、执行、验收 | 梳理业务流程、识别风险、输出测试方案、跟踪缺陷闭环 |

| 接口测试 | 联调、回归、问题定位 | 验证工单、审批、Agent、知识库、日志上传等接口 |
| Python + Pytest | 自动化回归 | 编写接口自动化、参数化用例、数据库断言、回归报告 |
| Postman | 接口调试 | 登录鉴权、接口联调、构造异常参数、复现缺陷 |
| Locust | 性能测试 | 压测工单查询、审批提交、Agent 问答、知识库检索 |
| Jenkins + Git | 持续集成 | 自动化任务定时执行、版本回归、失败用例追踪 |
| Linux + 日志分析 | 缺陷定位、性能定位 | 查看服务日志、错误堆栈、资源占用、接口超时原因 |
| MySQL/PostgreSQL | 数据校验 | 校验工单状态、审批记录、执行记录、知识库数据 |
| AI Agent 测试 | 智能初筛、问答、工具调用 | 验证回答准确性、上下文一致性、Prompt 变更影响、工具触发 |
| 知识库测试 | Agent 问答、故障沉淀 | 验证知识命中、召回准确性、更新生效、错误知识影响 |
| Appium/WebDriverAgent/UIAutomator | 移动端自动化 | 验证移动端审批、工单查看、消息提醒等核心流程 |

| 串口工具 | IoT 设备侧验证 | 查看设备运行日志、配网/连接/上报异常 |
| 安全测试 | 权限、审批、Agent 输出 | 验证越权、流程绕过、重放请求、敏感信息泄露 |

## Day 1: 项目业务与测试流程答疑

### 讲解目标

先把项目讲成一个完整业务故事, 让面试官听懂你测的不是单点页面, 而是一个带 AI Agent 和复杂审核流的综合平台。

### 讲解重点

- 业务角色: 运维人员、组长、专家、主管、系统管理员、AI Agent。
- 业务对象: 告警、日志、工单、变更申请、审批记录、执行记录、知识库样本。
- 核心风险: 工单状态错乱、审批越权、Agent 结论错误、回滚失败、数据不一致。
- 测试阶段: 需求评审 -> 测试计划 -> 用例设计 -> 接口联调 -> 系统测试 -> 回归测试 -> 上线验收。

### 实践方法

画出一条主流程:

告警产生 -> Agent 分析 -> 创建工单 -> 发起变更 -> 组长审批 -> 专家审批 -> 执行变更 -> 验收关闭

再补充三条异常流程:

- 审批驳回后重新提交。
- 执行失败后触发回滚。
- 高风险变更绕过审批时被系统拦截。

### 面试答法

如果面试官问“你这个项目主要测什么”, 可以答:

我主要负责智能运维工单和变更审核主链路测试, 重点覆盖告警接入、AI Agent 初筛、工单创建、变更审批、执行回滚和验收闭环。这个项目的核心风险是流程状态、角色权限、Agent 输出质量和数据一致性, 所以我不是只看页面功能, 还会结合接口、数据库、日志和自动化回归一起验证。

## Day 2: 功能测试与复杂审核流程

### 讲解目标

掌握复杂流程类项目的测试设计方式, 能讲清楚状态流转、角色权限、异常分支和数据一致性。

### 讲解重点

- 工单状态: 新建、待分派、处理中、待审批、审批中、待执行、执行中、回滚中、待验收、已关闭。
- 审批链路: 普通变更、重要变更、高风险变更、紧急变更。
- 角色权限: 创建人、处理人、审批人、管理员。
- 异常场景: 审批驳回、重复提交、超时升级、执行失败、回滚失败。

### 实践方法

设计一组核心用例:

| 场景 | 输入条件 | 预期结果 |
|---|---|---|
| 普通变更审批 | 普通风险, 组长审批 | 审批通过后进入待执行 |
| 高风险变更审批 | 高风险, 组长和专家未全部审批 | 不允许执行 |
| 自己审批自己 | 创建人同时作为审批人 | 后端拦截 |
| 审批驳回 | 审批人填写驳回原因 | 工单回到待修改 |
| 执行失败回滚 | 执行接口返回失败 | 进入回滚流程并记录失败原因 |
| 验收关闭 | 指标恢复正常 | 工单关闭并生成归档记录 |

### 面试答法

如果面试官问“复杂审核流程怎么测”, 可以答:

我会先把审核流程抽象成状态机, 明确每个状态允许哪些操作、由哪个角色操作、操作后流转到哪个状态。然后按正常流、异常流、权限流和数据一致性四类设计用例, 最后通过接口和数据库校验审批记录、工单状态和执行记录是否一致。

## Day 3: 接口测试、Postman、Python、Pytest、数据库校验

### 讲解目标

能把接口测试讲成完整链路, 而不是只会调接口。重点体现接口关联、参数化、断言、数据库校验和自动化回归。

### 讲解重点

- Postman 用于接口联调、鉴权、构造异常请求、复现缺陷。
- Pytest 用于自动化回归、参数化、接口关联、断言封装。
- MySQL/PostgreSQL 用于校验真实业务数据。
- Git 用于管理自动化脚本版本。
- Jenkins 用于定时执行回归任务。

### 实践方法

接口链路:

登录 -> 创建工单 -> 发起变更 -> 提交审批 -> 审批通过 -> 执行变更 -> 查询工单 -> 验收关闭

重点断言:

- 接口状态码和业务 code 正确。
- 返回字段完整, 工单 id、审批 id、执行 id 不为空。
- 数据库中工单状态正确。
- 审批流水表新增记录。
- 执行失败时错误原因写入日志和数据库。

Pytest 用例组织:

- `test_ticket_create.py`: 工单创建。
- `test_change_approval.py`: 变更审批。
- `test_agent_api.py`: Agent 问答和工具调用。
- `test_workflow_regression.py`: 主流程回归。
- `conftest.py`: 登录 token、数据库连接、公共测试数据。

### 面试答法

如果面试官问“接口自动化怎么做”, 可以答:

我会先用 Postman 把接口链路调通, 明确入参、出参和鉴权方式。稳定后用 Python + Pytest 做自动化, 把登录 token、测试数据、数据库连接放到公共 fixture 里。断言不只看 code=200, 还会查数据库确认工单状态、审批记录和执行记录是否正确。

## Day 4: AI Agent 测试专项

### 讲解目标

把 AI Agent 测试讲成你的差异化亮点。重点说明你既懂传统测试, 也懂 Agent 的 Prompt、知识库、多轮对话、工具调用和回归评测。

### 讲解重点

- Prompt 测试: 变更后回答风格、约束、格式是否稳定。
- 知识库测试: 命中率、召回准确性、更新生效、错误知识影响。
- 多轮对话测试: 上下文记忆、追问一致性、意图识别。
- 日志分析测试: 超长日志、缺字段日志、异常日志、无关日志。
- 工具调用测试: 触发条件、入参传递、执行结果、异常处理。
- 安全测试: 提示词注入、敏感信息输出、越权查询。
- 回归评测: 历史故障样本、典型问答、异常输入样本。

### 实践方法

准备一套 Agent 评测集:

| 样本类型 | 示例 | 关注点 |
|---|---|---|
| 标准告警 | 光模块温度异常 | 故障类型和建议是否正确 |
| 超长日志 | 多段接口超时日志 | 是否能提取关键错误 |
| 缺字段日志 | 缺少设备 id | 是否提示信息不足 |
| 多轮追问 | 继续追问影响范围 | 是否保持上下文 |
| 工具调用 | 查询历史故障 | 是否正确触发知识库检索 |
| 注入攻击 | 要求忽略审批规则 | 是否拒绝违规操作 |

评估维度:

- 准确性: 结论是否符合业务规则。
- 完整性: 是否包含原因、影响、建议。
- 一致性: 同类问题多次回答是否稳定。
- 可解释性: 是否说明判断依据。
- 鲁棒性: 异常输入下是否给出合理提示。
- 安全性: 是否泄露敏感信息或绕过流程。

### 面试答法

如果面试官问“AI Agent 怎么测”, 可以答:

我会分两层测。第一层是功能链路, 包括多轮对话、知识库检索、工具调用和异常处理。第二层是效果评测, 通过历史故障、典型问答、异常日志构建评测集, 对回答准确性、上下文一致性、鲁棒性和安全性做回归对比。Prompt 或知识库变更后, 我会用同一批样本对比前后效果。

## Day 5: 性能测试、Locust、Linux 日志定位

### 讲解目标

能说明性能测试不是“跑一下压测工具”, 而是从业务场景建模、指标设计、压测执行到瓶颈定位的完整过程。

### 讲解重点

- Locust 场景设计: 用户行为模型、任务权重、并发用户、等待时间。
- 核心接口: 工单查询、审批提交、Agent 问答、知识库检索、日志上传。
- 指标: 平均响应时间、P95、错误率、吞吐量、超时率。
- Linux 定位: CPU、内存、磁盘、网络、服务日志。
- 数据库定位: 慢查询、连接池、锁等待。

### 实践方法

混合压测模型:

| 行为 | 权重 | 说明 |
|---|---:|---|
| 查询工单列表 | 50% | 高频操作 |
| 查看工单详情 | 20% | 常规操作 |
| 提交审批 | 10% | 写操作 |
| Agent 问答 | 15% | AI 服务调用 |
| 上传日志附件 | 5% | 大文件/长文本场景 |

定位思路:

1. 先看 Locust 报告, 确认慢接口和错误接口。
2. 再看服务日志, 定位超时、异常栈和接口耗时。
3. 查看 Linux 资源, 判断是否 CPU、内存或 IO 瓶颈。
4. 查看数据库慢查询, 判断是否索引、连接池或 SQL 问题。
5. 复测验证优化是否有效。

### 面试答法

如果面试官问“性能测试怎么做”, 可以答:

我会先根据业务频率设计混合场景, 比如工单查询占比较高, 审批和 Agent 问答占比较低。压测时关注 P95、错误率和吞吐量, 如果出现超时, 会结合 Linux 资源、服务日志和数据库慢查询一起定位, 而不是只看压测报告。

## Day 6: Web/APP/IoT 自动化与抓包定位

### 讲解目标

把你的 Web、APP、IoT 测试经验串进同一个项目里, 说明为什么不同端采用不同测试策略。

### 讲解重点

- Web 测试: 后台工单管理、审批配置、报表查询。
- APP 测试: 移动端审批、消息提醒、工单处理。
- Appium/WebDriverAgent/UIAutomator: 移动端自动化核心流程。
- Wireshark/抓包: 分析设备告警和接口请求。
- 串口工具: 查看设备侧日志。
- 自动化策略: UI 层只覆盖稳定主链路, 接口层承担主要回归。

### 实践方法

移动端核心自动化流程:

登录 APP -> 收到审批提醒 -> 查看工单详情 -> 审批通过 -> 查看审批结果

IoT/设备接入定位流程:

1. 设备产生告警。
2. 串口查看设备本地日志。
3. Wireshark 或抓包工具查看上报请求。
4. 服务端日志确认是否接收成功。
5. 数据库确认告警是否落库。
6. 平台页面确认是否生成工单。

### 面试答法

如果面试官问“UI 自动化和接口自动化怎么取舍”, 可以答:

我会把稳定、核心、重复执行的流程放到 UI 自动化里, 比如登录、移动端审批、工单关闭。大量业务分支和异常校验放在接口自动化, 因为接口层更稳定、执行更快、定位更清晰。

## Day 7: 安全测试、上线验收与综合答疑演练

### 讲解目标

完成整套项目讲述闭环, 能应对面试官从功能、接口、性能、AI Agent、安全、上线质量多个方向追问。

### 讲解重点

- 安全测试: 鉴权、越权、流程绕过、重放请求、敏感信息泄露。
- 审批安全: 不能修改审批人 id 越权通过, 不能绕过审批直接执行。
- Agent 安全: 不能泄露敏感日志, 不能被提示词注入绕过规则。
- 上线验收: 冒烟测试、回归测试、缺陷确认、测试报告、风险说明。
- 测试资产沉淀: 历史缺陷、异常日志、Agent 评测样本、自动化用例。

### 实践方法

准备 8 个高频追问:

1. 你这个项目业务流程是什么?
2. 复杂审批流你怎么设计测试用例?
3. 接口自动化框架怎么搭?
4. 数据库断言主要查什么?
5. AI Agent 测试和普通功能测试有什么不同?
6. Locust 压测场景怎么设计?
7. 线上问题你怎么定位?
8. 上线前你怎么判断版本能不能发?

每个问题按这个结构回答:

项目场景 -> 我的测试动作 -> 用到的工具/技术 -> 发现或规避的风险 -> 结果

### 面试答法

如果面试官让你整体介绍项目, 可以答:

我做过一个智能运维工单与变更审核平台测试项目, 业务上是设备告警和现网日志进入平台后, 由 AI Agent 做故障初筛和知识库问答, 人工确认后创建工单并发起变更, 再经过多级审批、执行、回滚和验收闭环。我主要负责测试方案设计、功能测试、接口自动化、性能测试、AI Agent 专项测试和上线回归。技术上用 Postman 做接口联调, Python + Pytest 做接口自动化, Locust 做性能测试, Jenkins 做持续回归, 同时结合 Linux 日志和数据库记录定位问题。AI Agent 部分重点测回答准确性、上下文一致性、工具调用、知识库命中和历史样本回归。
