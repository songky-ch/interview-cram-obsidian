## 总结

今天整体比 Day1 更工程化, 但在接口链路依赖、查询接口断言、测试数据清理和 CI 偶现失败定位上还需要更贴近 Pytest 和接口自动化实践。

## 易错点 1: 把接口链路拆成互相依赖的多个 test

错误表现:

- 认为“登录 -> 创建设备 -> 绑定设备 -> 上报属性 -> 查询状态”应该拆成多个互相依赖的 test。
- 理由是方便定位, 但忽略 Pytest 用例独立性。

修正方式:

- 单接口能力拆成独立 test。
- 端到端链路写成一个场景用例, 用 Allure step 或日志拆清每一步。
- 前置数据用 fixture 准备, 不依赖上一个 test 的执行结果。

推荐表达:

```text
Pytest 用例之间尽量独立, 不应该依赖执行顺序。单接口测试可以拆成独立 test, 端到端链路可以作为一个场景用例, 但用 Allure step 或日志记录每一步请求、响应和断言。
```

## 易错点 2: 查询接口断言只讲断言语法

错误表现:

- 只说状态码断言、包含断言、不等断言、JSON 等值断言、异常断言。
- 没有说报表列表查询接口具体断言哪些业务内容。

修正方式:

- 查询类接口断言要分层: HTTP 状态、业务 code、JSON 结构、业务字段、分页排序、权限范围、数据库结果、异常返回。

推荐表达:

```text
以报表列表查询接口为例, 我会断言 HTTP 状态、业务 code、total/list/pageSize/pageNum 结构、字段类型、筛选条件是否生效、分页排序是否正确、权限范围是否正确, 并用数据库或下游查询接口校验统计结果。
```

## 易错点 3: 数据清理主要说 teardown

错误表现:

- 说用 tear_down 做后置清理。
- 没有突出 Pytest fixture / yield。

修正方式:

- 用 fixture 管理测试数据生命周期。
- fixture 前置创建运行时数据, `yield` 给用例使用, yield 后删除数据或记录清理失败 ID。

推荐表达:

```text
在 Pytest 里, 我会用 fixture 管理数据生命周期。前置创建测试设备或报表数据, yield 给用例使用, 用例结束后在 fixture 后置逻辑里调用删除接口或数据库清理。
```

## 易错点 4: 环境隔离只说路径隔离

错误表现:

- 只说测试环境和正式环境的数据存放路径不同。
- 没有覆盖环境地址、数据库、账号、租户、数据前缀和生产保护。

修正方式:

- 环境隔离应通过配置区分 base_url、数据库连接、账号、租户、设备前缀和数据目录。
- 自动化默认禁止在生产环境执行写接口。

推荐表达:

```text
环境隔离不只是文件路径隔离, 还包括 base_url、数据库、账号、租户、数据标识和权限范围隔离。生产环境写接口默认禁用或加保护。
```

## 易错点 5: CI 偶现失败定位缺少失败现场

错误表现:

- 只对比本地和 Jenkins 配置、数据、依赖版本。
- 提到打断点, 但 CI 场景更需要日志和报告证据。

修正方式:

- 保留 Jenkins 构建号、执行时间、请求参数、响应体、错误码、traceId、测试数据 ID、Allure 报告。
- 再排查环境、数据、并行执行、脚本断言、服务端日志。

推荐表达:

```text
我会先看 Jenkins 失败现场, 包括构建号、执行时间、请求参数、响应体、traceId 和测试数据 ID, 再对比环境配置、数据状态、并行执行情况, 最后结合服务端日志判断是否是接口偶现缺陷。
```

## 易错点 6: 发布后补跑全量回归表述要谨慎

错误表现:

- 容易表达成“先发布, 30 分钟后全量结果出来再决定”。
- 面试官可能认为高风险未知问题被放到发布后才发现。

修正方式:

- 发布前必须先跑核心冒烟、高风险变更、历史缺陷和影响链路。
- 发布后可以继续全量补跑和线上监控, 但要明确阻塞发布标准和回滚条件。

推荐表达:

```text
时间不足时我会先跑核心冒烟集、高风险变更集和历史缺陷集。核心链路、鉴权、权限、数据准确性或 500 超时失败属于阻塞发布。全量回归可继续异步补跑, 发布后配合线上只读验证和监控。
```

## 易错点 7: 参数化外部数据的价值表达不准

错误表现:

- 把 YAML 外部数据的主要价值说成权限和安全隔离。

修正方式:

- 外部数据的核心价值是数据和代码分离、维护方便、用例评审清晰、同一测试逻辑复用多组数据。
- 敏感账号密码不应写在 YAML 中。

推荐表达:

```text
YAML 数据驱动的价值是数据和代码分离, 新增状态、角色和时间范围时不需要改测试逻辑。敏感账号密码不写在 YAML 中, 应通过环境变量或配置读取。
```

## 易错点 8: 字段变更断言要区分新增和破坏性变更

错误表现:

- 只说新增 / 删除字段要重新校验, 没有明确风险等级。

修正方式:

- 新增字段通常不应导致旧用例失败。
- 删除字段、字段类型调整、枚举变化属于可能破坏兼容性的变更, 必须确认接口契约和调用方影响。

推荐表达:

```text
核心字段强断言, 非核心字段用 JSON Schema 做结构和类型校验。新增字段通常不让旧用例失败, 但删除字段、类型调整和枚举变化要触发接口契约和关联接口回归。
```

## 复习口令

```text
接口自动化回答要落到项目链路、Pytest 机制、业务断言、数据生命周期和 CI 失败证据。
```
