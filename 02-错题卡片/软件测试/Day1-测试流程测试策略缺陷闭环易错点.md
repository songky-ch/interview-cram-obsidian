## 总结

今天的主要问题不是方向错误, 而是回答偏列点, 项目化表达、定位闭环和量化验收标准不够强。

## 易错点 1: AI Agent 测试回答太像通用理论

错误表现:

- 只说准确性、幻觉、异常处理。
- 没有带出睿思 AI Agent 的现网日志分析、分析件初筛、业务问答、多轮对话、工具调用。

修正方式:

- 先说项目场景, 再说测试范围和风险。
- 用“输入样本 -> Agent 分析过程 -> 工具调用 -> 输出结论 -> 回归评测”组织回答。

推荐表达:

```text
以睿思 AI Agent 的现网日志分析需求为例, 我会先明确输入日志、知识库、Prompt、工具调用和最终业务结论, 再围绕回答准确性、幻觉、上下文一致性、异常处理和工具调用边界设计测试。
```

## 易错点 2: 回归测试数据集只说输入, 不说评估标准

错误表现:

- 只说正向、负向、边界、大文件、历史缺陷。
- 没有说明标准答案、关键要点、禁止项、工具调用预期。

修正方式:

- AI Agent 回归数据集必须同时维护输入和评估规则。
- 每条样本应包含预期结论、关键依据、禁止出现的幻觉内容和工具调用预期。

推荐表达:

```text
这类回归数据集不是只保存输入样本, 还要保存标准答案、关键要点、禁止项和工具调用预期, 这样才能判断表达不同但语义正确的回答是否通过。
```

## 易错点 3: Prompt 变更回归范围说得太绝对

错误表现:

- 直接说“仍然需要完整回归”。
- 没有说明如何控制回归成本。

修正方式:

- Prompt 变更必须回归, 但要按影响面分层。
- 核心链路、历史缺陷、高风险样本必须跑；低风险场景可以抽样。

推荐表达:

```text
我不会因为只改 Prompt 就降低风险, 但也不会机械全量。会先做 Prompt diff 分析, 看影响目标、上下文、约束、输出格式还是工具调用说明, 再决定核心场景必跑、受影响场景重点跑、低风险场景抽样。
```

## 易错点 4: 平台测试方案只列功能点, 不讲业务链路

错误表现:

- 直接列数据转发、驳回、权限、报表展示。
- 缺少从数据产生到报表展示的端到端链路。

修正方式:

- 先讲业务链路, 再按功能、权限、接口、数据库、报表、联动模块拆测试。

推荐表达:

```text
我会先把需求拆成端到端链路: 数据创建、提交、审核、驳回、重新提交、处理完成、统计入报表。每一步分别验证状态、权限、接口返回、数据库记录和下游模块展示。
```

## 易错点 5: 数据不一致定位链路不完整

错误表现:

- 只怀疑前端分页、字段映射、渲染问题。
- 没有覆盖请求参数、数据库、统计口径、权限范围、缓存。

修正方式:

- 固定现象后逐层定位: 页面筛选条件 -> 请求参数 -> 接口响应 -> 前端渲染 -> 数据库 -> 统计口径 -> 权限 -> 缓存 / 异步同步。

推荐表达:

```text
我会先确认不一致是数量、字段、排序还是范围问题, 再对比页面筛选条件和接口请求参数。如果接口正确但页面错误, 偏前端渲染；如果接口和数据库不一致, 查后端查询逻辑、统计口径或缓存同步。
```

## 易错点 6: MySQL 验证只停留在 count

错误表现:

- 只说把筛选条件代入 SQL。
- 只说 select count(*) 或 select 字段。

修正方式:

- 统计口径问题要拆 count、group by、sum / avg、join、distinct、权限范围和数据时效性。

推荐表达:

```text
如果是数量问题, 我会用 count 验证总数；如果是分类问题, 用 group by 拆状态或类型；如果是成功率或耗时, 拆分子和分母；如果涉及多表, 重点看 join 是否导致重复统计, 是否需要 distinct。
```

## 易错点 7: 争议缺陷处理表达偏弱

错误表现:

- 说“等用户反馈增加后回溯”。
- 说“让用户适应”。

修正方式:

- 用需求、验收标准和用户影响做证据化沟通。
- 推动产品、开发、测试共同确认风险和上线决策。

推荐表达:

```text
我不会先争论它是不是 bug, 而是拿需求文档、验收标准和用户影响做证据对齐。如果最终确认为设计如此, 我会记录风险、明确影响范围和触发条件, 并推动产品确认上线决策或补充优化措施。
```

## 易错点 8: 完整项目流程缺少项目开头

错误表现:

- 直接说需求分析、测试设计、缺陷跟踪。
- 没有说“以哪个项目、哪个需求为例”。

修正方式:

- 第一秒绑定真实项目。
- 使用“项目背景 -> 需求分析 -> 测试方案 -> 测试执行 -> 缺陷闭环 -> 回归验收 -> 结果沉淀”结构。

推荐表达:

```text
以睿思 AI Agent 的现网日志分析需求为例, 我会先参加需求评审, 明确这次需求是新增日志分析能力还是优化业务问答效果, 然后分析它和 Prompt、知识库、工具调用及历史高频场景的耦合关系。
```

## 复习口令

```text
回答任何项目题时, 先绑定项目, 再讲链路, 再讲分层验证, 最后讲缺陷闭环和验收标准。
```
