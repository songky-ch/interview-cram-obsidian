## 总结

今天在接口 500 定位上表现较好, 能使用时间戳、grep、tail、MySQL 死锁信息。但 Linux 命令场景化表达、NullPointerException 深挖、数据准确性口径和权限数据定位还需要加强。

## 易错点 1: NullPointerException 定位太粗

错误表现:

- 只说 grep 后拿完整日志。
- 只说本地 debug, 没有说明如何缩小空对象来源。

修正方式:

- 查完整堆栈、traceId、接口路径、业务类、方法名和行号。
- 结合入参、数据库查询结果、关联表、依赖服务返回、配置和缓存判断空对象来源。

推荐表达:

```text
我会先按 traceId 或时间点拿到完整堆栈, 看最上层业务类、方法和行号, 再结合入参和数据库状态判断是必填参数缺失、查询结果为空、关联表缺失、依赖服务返回空对象, 还是配置或缓存未初始化。
```

## 易错点 2: Jenkins 批量回归 500 排查范围偏窄

错误表现:

- 主要怀疑死锁和接口延迟导致前置数据未就绪。

修正方式:

- 补充用例依赖、数据污染、执行顺序变化、共享账号、共享数据、接口限流、线程池 / 连接池不足、Jenkins 环境差异。

推荐表达:

```text
Jenkins 批量才失败时, 我会优先看并发和数据问题, 包括死锁、唯一键冲突、用例顺序变化、清理逻辑误删、异步任务未完成、限流、连接池耗尽和 Jenkins 环境差异。
```

## 易错点 3: 报表数据准确性验证缺少完整口径

错误表现:

- 主要说 `select count(*)`。
- 只提逻辑删除、联表和 schema 环境。

修正方式:

- 统一页面、接口、SQL 的时间范围、状态、部门、角色、租户、权限范围、逻辑删除、数据来源和统计时间点。
- 用 count、group by、sum、join 去重、汇总表 / 缓存 / 定时任务一起验证。

推荐表达:

```text
我会先统一筛选条件和权限范围, 再用 count 查总数、group by 查状态分布, 如果有 join 就验证是否一对多放大, 如果报表来自汇总表或缓存, 还要确认定时任务和刷新时间。
```

## 易错点 4: 多表 join 重复统计只想到 distinct

错误表现:

- 只说看有没有 distinct。

修正方式:

- 先确认主统计对象。
- 逐步加 join 看数量在哪一步放大。
- 用主表 id 分组 `having count(*) > 1` 找重复。
- 根据场景使用 `count(distinct main.id)`、`exists`、子表预聚合或修正 join 条件。

推荐表达:

```text
我会先单查主表数量, 再逐步加 join, 看哪张子表导致主表 id 重复。可以按主表 id group by 并 having count(*) > 1 找出被放大的记录, 再判断是 count distinct、exists、子表预聚合还是 join 条件缺失。
```

## 易错点 5: Linux 命令回答只列命令

错误表现:

- 只列 tail、grep、find、cd、cat、vim。
- 没有说明命令适用场景。
- 把 vim 编辑服务器文件当成常规排查动作。

修正方式:

- 按场景回答: 实时日志、关键字过滤、大文件查看、历史压缩日志、进程端口、系统资源、配置只读确认。

推荐表达:

```text
接口可复现时用 tail -f 实时看日志；查异常用 grep 按 traceId、时间点和 ERROR 过滤；大日志用 less；历史压缩日志用 zgrep；服务不可用查 ps 和 netstat；响应慢查 top、free -m、df -h；配置问题只读 grep, 不随意 vim 修改。
```

## 易错点 6: 接口慢定位证据链不够完整

错误表现:

- 只说 top、free、df、手工执行 SQL、netstat。

修正方式:

- 先通过 traceId / 接口耗时日志定位慢在哪一段。
- 服务侧看资源和线程 / GC 风险。
- SQL 侧看慢查询、EXPLAIN、扫描行数、锁等待。
- 依赖侧看下游调用日志和耗时。
- 网络侧用 curl -w、ping、telnet/nc 或连接状态辅助判断。

推荐表达:

```text
我会先用 traceId 看接口总耗时拆分, 判断慢在业务处理、数据库、下游依赖还是连接阶段。然后分别看 Linux 资源、慢 SQL / EXPLAIN、下游调用日志和网络连通性。
```

## 易错点 7: EXPLAIN 字段掌握不完整

错误表现:

- 只说 type、key、key_len、rows。

修正方式:

- 补充 possible_keys、filtered、Extra、id / select_type。
- 测试侧表达为识别慢 SQL 风险并提供证据, 不越界承诺索引优化方案。

推荐表达:

```text
EXPLAIN 我会看 type 判断访问方式, possible_keys 和 key 判断索引是否可用及实际命中, key_len 判断联合索引用到几段, rows 和 filtered 判断扫描与过滤成本, Extra 重点看 Using filesort、Using temporary、Using index。
```

## 易错点 8: 权限查不到数据时首要怀疑点偏了

错误表现:

- 过早怀疑锁表。

修正方式:

- 先查用户、角色、部门、租户、数据范围、前端查询条件、业务数据状态、逻辑删除、权限缓存、SQL 过滤和页面展示。

推荐表达:

```text
用户有权限但查不到数据时, 我会先确认账号、角色、部门、租户和数据范围, 再查前端参数、业务数据状态、逻辑删除、SQL where / join 和权限缓存。锁表一般不是首要怀疑点。
```
